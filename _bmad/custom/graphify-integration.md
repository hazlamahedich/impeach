# Graphify Integration for BMad

This project maintains a live knowledge graph of its planning artifacts, research, architecture, and UX designs. The graph is generated by [graphify](https://github.com/safishamsi/graphify) and refreshed as documents evolve.

## Where the Graph Lives

```
graphify-out/
  graph.json          - GraphRAG-ready graph data (nodes, edges, communities)
  GRAPH_REPORT.md     - Plain-language audit report (god nodes, surprising connections, questions)
  graph.html          - Interactive browser visualization
```

## How to Query It

### CLI (during a BMad workflow)

```bash
graphify query "<question>"              # BFS traversal - broad context
graphify query "<question>" --dfs        # DFS traversal - trace a specific path
graphify path "<concept A>" "<concept B>" # shortest path between two concepts
graphify explain "<concept>"              # plain-language explanation of a node
graphify --update                          # incremental refresh after file changes
```

### MCP (programmatic access for agents)

The `graphify` MCP server is registered in opencode and exposes these tools:

- `query_graph` - ask a natural-language question against the graph
- `get_node` - retrieve a specific node
- `get_neighbors` - get connected nodes
- `get_community` - get all nodes in a community
- `god_nodes` - get the most central nodes
- `graph_stats` - get graph statistics
- `shortest_path` - find path between two nodes

## Workflow-Specific Guidance

### `bmad-document-project`

When documenting this brownfield project:

1. Read `graphify-out/GRAPH_REPORT.md` first to understand community structure and god nodes.
2. Use `graphify query` to trace how concepts connect across documents.
3. Surface surprising connections and knowledge gaps as documentation priorities.

### `bmad-investigate`

When investigating a bug, incident, or unfamiliar area:

1. Use `graphify query "how does X relate to Y"` to trace dependencies and relationships.
2. Use `graphify path "<symptom>" "<root area>"` to find the shortest conceptual path.
3. Cite graph edge `source_file` when recording confirmed evidence in the case file.

### `bmad-create-architecture`

When creating or revising architecture:

1. Run `graphify --update` to refresh the graph from the latest documents.
2. Use the graph to understand existing architectural decisions and their rationale.
3. Cross-reference proposed changes against the community hubs in GRAPH_REPORT.md.

### `bmad-generate-project-context`

When generating `project-context.md`:

1. Treat `graphify-out/GRAPH_REPORT.md` as a condensed map of the project.
2. Pull the god nodes and community summaries into the context document.
3. Include the graph's surprising connections as "things every agent should know."

### `bmad-quick-dev`

Before implementing a change:

1. Query the graph for the feature/area you are changing: `graphify query "<feature>"`.
2. Identify bridge nodes (high betweenness) that may indicate blast radius.
3. Check the shortest path from the change area to security, citation, or trust-framework nodes.

### `bmad-code-review`

During code review:

1. Use the graph to verify that changed concepts still connect correctly to architecture, epics, and trust invariants.
2. Cross-check new dependencies against the existing community structure.
3. Flag any change that severs a connection to a god node without explicit rationale.

## Keeping the Graph Fresh

Run these after significant document changes:

```bash
graphify --update                          # incremental - only changed files
graphify . --mode deep                     # full re-extraction with richer inferred edges
graphify . --watch                         # continuous incremental updates in background
```

## Confidence & Audit

Every graph edge is tagged:

- **EXTRACTED** - explicit relationship found in source (confidence 1.0)
- **INFERRED** - reasonable inference (confidence 0.4-0.9)
- **AMBIGUOUS** - uncertain, flagged for review (confidence 0.1-0.3)

Treat INFERRED edges as leads, not facts. Verify against source files before depending on them in architecture or implementation decisions.
